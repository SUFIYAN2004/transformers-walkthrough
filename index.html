A walk through the architecture

Transformers

01 / 18

Input layer

What is The entry point of the model. Raw text is split into tokens, and every token is mapped to a vector of numbers — an embedding — that the network can actually compute with.
why we use Neural networks only understand numbers. Embeddings turn words into points in space where similar meanings end up close together.
how its works A tokenizer breaks the sentence into sub-word pieces, each piece is looked up in an embedding table, and the result is a matrix of shape [tokens × dimensions] ready for the next stage.
Example
"the cat sat" → ["the", "cat", "sat"]
"cat" → [0.12, -0.48, 0.91, ... 512 dims]
02 / 18

Positional encoding

What is A pattern of values added to each token's embedding that encodes where that token sits in the sequence.
why we use Unlike RNNs, a transformer looks at all tokens at once — it has no built-in sense of order. Without this step, "dog bites man" and "man bites dog" would look identical.
how its works Sine and cosine waves of different frequencies are computed for each position and added directly to the token embeddings, giving every position a unique, consistent signature.
Example
03 / 18

Self-attention

What is The mechanism that lets every token look at every other token and decide how much each one matters, using Query, Key, and Value vectors.
why we use Meaning depends on context. In "the bank of the river," attention lets "bank" pull meaning from "river" directly, no matter how far apart they are.
how its works Each token's Query is compared against every token's Key to get a score, scores are turned into weights with softmax, and those weights blend the Value vectors into a new, context-aware representation.
Example — hover a word
The cat sat on the mat
Line thickness ≈ attention weight from the hovered token
04 / 18

Multi-head attention

What is Several attention mechanisms running in parallel, each with its own learned Query/Key/Value projections — its own "head."
why we use One attention pattern can only capture one kind of relationship at a time. Multiple heads let the model track grammar, meaning, and reference simultaneously.
how its works The embedding is split into several smaller chunks, each chunk runs its own attention independently, and the results are concatenated and passed through one more linear layer.
Example
HEAD 1 "cat" ↔ "sat" — tracks subject → verb
HEAD 2 "the" ↔ "mat" — tracks article → noun
HEAD 3 "sat" ↔ "on" — tracks verb → preposition
05 / 18

Feed-forward network

What is A small two-layer neural net applied to each token position on its own, right after attention.
why we use Attention mostly mixes and re-weights information between tokens. The FFN is where the model actually does non-linear processing on each token's representation.
how its works Each token vector is expanded to a much larger hidden size, passed through a non-linearity like GELU, then projected back down to the original size — the same two layers, reused for every position.
Example
512 dims → linear + GELU → 2048 dims → linear → 512 dims
06 / 18

Encoder

What is A stack of identical layers that reads the whole input at once and builds a rich, context-aware representation of it.
why we use One pass of attention isn't enough to capture deep relationships. Stacking layers lets the model refine its understanding step by step.
how its works Each layer runs multi-head self-attention, adds the result back to its input (residual connection) and normalizes it, then does the same with a small feed-forward network.

07 / 18

Decoder

What is A second stack of layers that generates output one token at a time, using both what it has generated so far and the encoder's representation of the input.
why we use Tasks like translation or chat need to produce a sequence, not just understand one — the decoder is what actually writes the output.
how its works Masked self-attention stops a token from seeing future tokens, cross-attention lets it consult the encoder's output, and a feed-forward layer refines the result — then a final softmax picks the next word.
Example — one decoder layer

Add & Normalize residual
Cross-Attention reads encoder output
Add & Normalize residual
Feed-Forward Network Nx
Add & Normalize residual
stacked N times → Linear → Softmax → next token
08 / 18

Output layer

What is The final linear layer and softmax that turn the decoder's last hidden vector into a probability over every word in the vocabulary.
why we use Somewhere the model has to commit to an actual next word, out of tens of thousands of possibilities — this is that step.
how its works A linear layer projects the hidden vector to vocabulary-size logits, softmax turns those into probabilities, a token is picked (greedily, or sampled with a temperature), and it's fed back in to generate the next one.
Example — sampling the next token
next_token.pyPyTorch
logits = model(hidden_state) # [vocab_size] probs = torch.softmax(logits / temperature, dim=-1) next_token = torch.multinomial(probs, num_samples=1) # "The cat sat on the ___" → probs: mat 0.41, floor 0.18, chair 0.09, ...
09 / 18

Three shapes

What is Not every transformer uses both encoder and decoder — three shapes emerged, each suited to different tasks.
why we use Understanding text, generating text, and transforming one sequence into another need different information flow — so the field split into specialized shapes.
Example

Encoder-only

BERT. Reads the whole input at once (bidirectional). Best for classification, search, embeddings.

Decoder-only

GPT. Reads left-to-right, predicts one token at a time. Best for open-ended generation, chat.

Encoder–Decoder

T5. Encoder understands input, decoder writes output. Best for translation, summarization.

Fine-tuning

What is Taking a pretrained transformer and continuing to train it — usually a small part of it — on a narrower, task-specific dataset.
why we use Pretraining teaches general language ability; fine-tuning teaches it your domain, your tone, or a specific task, without the cost of training from scratch.
how its works Full fine-tuning updates every weight. LoRA / QLoRA instead freeze the base model and train small low-rank "adapter" matrices injected into the attention layers — far cheaper, and easy to swap out.
Example — LoRA fine-tuning
Hugging Face Transformers PEFT bitsandbytes
finetune_lora.pyPEFT + Trainer

Prompt engineering

What is Shaping the input text to reliably get the behavior you want — without touching a single weight.
why we use The same model can perform very differently depending on how a task is phrased. It's the cheapest, fastest lever for improving results.
how its works A system message sets role and constraints, a few worked examples (few-shot) show the pattern, and asking the model to reason step by step (chain-of-thought) improves multi-step answers.
Example — few-shot + system prompt
prompt.pyChat API
messages = [ {"role": "system", "content": "Classify sentiment as positive, negative, or neutral. Reply with one word."}, {"role": "user", "content": "This movie was incredible."}, {"role": "assistant", "content": "positive"}, # few-shot example {"role": "user", "content": "The plot made no sense."}, ]

Retrieval-Augmented Generation

What is A pattern that looks up relevant documents from an external store and feeds them into the prompt, before the transformer generates its answer.
why we use A model's knowledge is frozen at training time and limited by context length. RAG lets it answer from live, private, or huge document sets without retraining.
how its works Documents are embedded and stored in a vector database. At query time, the question is embedded too, the closest chunks are retrieved by similarity, and both are handed to the transformer as context.
Example — RAG pipeline
LangChain Chroma / FAISS OpenAI / Claude
rag_pipeline.pyLangChain
from langchain_community.vectorstores import Chroma from langchain_openai import OpenAIEmbeddings, ChatOpenAI from langchain.chains import RetrievalQA vectordb = Chroma.from_documents(docs, OpenAIEmbeddings(), persist_directory="./db") retriever = vectordb.as_retriever(search_kwargs={"k": 4}) # top-4 chunks qa_chain = RetrievalQA.from_chain_type( llm=ChatOpenAI(model="gpt-4o"), retriever=retriever, return_source_documents=True, ) response = qa_chain.invoke({"query": "What's our refund policy?"}) print(response["result"])

Quantization

What is Storing a model's weights in fewer bits — say, 4-bit instead of 16-bit — to shrink it and speed it up.
why we use A full-precision model can be tens of gigabytes. Quantization lets the same model run on a consumer GPU, or even a laptop, with only a small quality trade-off.
how its works Weights are mapped from floating point into a small set of discrete levels using a calibration pass, then de-quantized on the fly during each matrix multiplication at inference time.
Example — loading a 4-bit model
bitsandbytes GGUF / llama.cpp
load_quantized.pybitsandbytes

Running it offline

What is Running a quantized model entirely on your own machine — no API, no internet — using a lightweight local inference engine.
why we use Privacy (nothing leaves your machine), zero API cost, no rate limits, and it works with no connection at all — useful for laptops, edge devices, or air-gapped environments.
how its works The model is converted to GGUF format and quantized to a chosen bit-width. llama.cpp then loads it and runs inference in pure C/C++ — on CPU, or offloaded to GPU layers if one is available.
Example — llama.cpp, quantized to Q3_K_M
llama.cpp Ollama LM Studio
terminalllama.cpp
# 1. convert the HF checkpoint to GGUF python convert_hf_to_gguf.py ./llama-3-8b --outfile llama3-8b-f16.gguf # 2. quantize it down — pick a bit-width / method ./llama-quantize llama3-8b-f16.gguf llama3-8b-Q3_K_M.gguf Q3_K_M # 3. run it, fully offline ./llama-cli -m llama3-8b-Q3_K_M.gguf -p "Explain attention in one sentence." -n 128

KV cache & headroom

What is A running store of every token's Key and Value vectors from attention, kept in memory so they never have to be recomputed while generating.
why we use Without it, producing token 500 would mean re-running attention over all 499 tokens before it — again. The cache turns that into an O(1) lookup per step instead of redoing the whole pass.
how its works Each token computes its Query, Key, and Value once. The Key and Value are appended to a per-layer cache; every future token attends to that growing cache instead of recomputing K/V for tokens already seen.
Example — enabling it, and what it costs
generate.pyTransformers
model.generate( input_ids, max_new_tokens=200, use_cache=True, # on by default — this is what makes generation fast )
WHY IT NEEDS HEADROOM
cache size ≈ 2 × layers × kv_heads × head_dim × seq_len × batch × bytes
Llama-3-8B, 8K context, batch 1, fp16 → ~4.3 GB just for the cache — on top of the ~16GB the weights already take. Double the context, double the cache; double the batch, double it again.
This is the "headroom" you plan for: VRAM left over after loading weights, sized for your longest context × batch size, not just the model file.

Agentic AI

What is Wrapping a transformer in a loop that lets it plan, call tools, observe results, and decide its own next step — rather than producing one reply and stopping.
why we use Many real tasks need multiple steps and outside actions — searching the web, running code, calling an API — not just a single generated answer.
how its works The model is given a set of tools and a goal. It reasons about what to do, calls a tool, reads the result, and repeats — often as a graph of steps — until the goal is met or a stopping condition is hit.
Example — a tool-using agent
LangGraph CrewAI AutoGen
agent.pyLangGraph
from langgraph.prebuilt import create_react_agent from langchain_core.tools import tool @tool def search_web(query: str) -> str: """Search the web and return top results.""" return web_search_api(query) @tool def run_python(code: str) -> str: """Execute python and return stdout.""" return sandboxed_exec(code) agent = create_react_agent(model="claude-sonnet-5", tools=[search_web, run_python]) result = agent.invoke({ "messages": [{"role": "user", "content": "Find last quarter's revenue and plot it"}] }) # agent decides: search_web → run_python → reply

Safety & guardrails

What is The layer of training and runtime techniques that keep a model's behavior aligned, safe, and inside intended bounds — separate from the core architecture.
why we use A capable model can also produce harmful, biased, or off-policy output. Raw next-token prediction alone doesn't know what it shouldn't say.
how its works RLHF or DPO further trains the base model on human preference data to shape its behavior, while runtime layers — content filters, system prompts, rate limits — catch what training alone doesn't.
Example
Training-timeRLHF / DPO shape the base model's preferences
RuntimeContent classifiers, system prompts, refusal handling
Product-levelRate limits, monitoring, human review loops

Where it ends up

Same encoder-decoder skeleton, reused everywhere something needs to be understood or generated.

Translation

The original use case — encoder reads one language, decoder writes another.

Chatbots

Decoder-only models (like GPT) generate replies token by token from a prompt.

Summarization

Encoder digests a long document, decoder writes a compressed version.

Code generation

Same next-token prediction, trained on code instead of prose.

Vision transformers

Images are cut into patches and treated as "tokens" for the same architecture.

Search & retrieval

Encoder-only models (like BERT) turn text into embeddings for semantic search.

That's the stack.

Input → position → attention → heads → encoder → decoder → output. Every large language model today is some variation on this.
